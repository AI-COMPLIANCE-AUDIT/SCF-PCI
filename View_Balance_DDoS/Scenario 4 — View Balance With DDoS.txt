**Scenario 4 — View Balance With DDoS 3.1**

The scenario begins at **S0_User_Logged_In**, the initial situation. At this state, the threat **T1_DDoS_Attack** enters, and the item **SessionID (xsd:string)** enters scope, confirming an active authenticated session. From S0, two paths immediately diverge — the legitimate balance-view path and the DDoS attack path.

The first path is the **legitimate balance-view path**. Action **a01_Choosing_Operation** leads to **S1_Operation_Chosen**, at which point the item **AccountID (xsd:string)** enters scope. Action **a12_Requesting_Balance** leads to **S2_Balance_Requested**. At S2, two events enter — **E1_Return_Balance** and **E2_Disconnect** — and the item **CurrentBalance (xsd:int)** enters scope. From S2, two paths diverge. The first is the forward path: action **a23_Showing_Balance** leads to **S3_Balance_Shown**, at which point event **E3_User_Logout** enters. Action **a34_Logging_Out** leads to the normal terminal state **S4_User_Logged_Out**. The second path from S2 is a disruption path caused by the DDoS attack: event **E2_Disconnect** triggers action **a28_Loosing_Connection_ABN**, leading to **S8_Disconnected_ABN**, which is addressed separately below.

The second path from S0 is the **DDoS attack path**. Action **a05_Sending_Packets** leads to **S5_Request_Packets_Sent_ABN**, representing the DDoS flood of malicious packets being launched. From S5, two paths diverge. The first path is triggered by event **E4_Network_Problem_ABN**: action **a56_Overloading_ABN** leads to **S6_Network_Crashed_ABN**, representing the network infrastructure collapsing under the packet flood. From S6, action **a68_Resulting_In_ABN** leads to **S8_Disconnected_ABN**. The second path from S5 does not require an event trigger: action **a57_Overloading_ABN** leads directly to **S7_Server_Crashed_ABN**, representing the server itself crashing under the overload. At S7, event **E6_Server_Fails_ABN** enters, and action **a78_Resulting_In_ABN** leads to **S8_Disconnected_ABN**.

**S8_Disconnected_ABN** is therefore the convergence state for three independent paths: from S2 via **a28** (connection loss during the legitimate balance query), from S6 via **a68** (network crash), and from S7 via **a78** (server crash). At S8, event **E5_Run_Load_Balancer** enters. From S8, two paths diverge. The first is the criminal terminal path: action **a81_Cancelling_Operation_ABN** leads to the abnormal terminal state **S10_Operation_Cancelled_ABN**, representing the DDoS attack successfully denying service with no mitigation applied. The second is the mitigation path: action **a89_Mitigating_By_COR** leads to **S9_Load_Balancer_Active**, representing the load balancer being activated to absorb and redistribute the attack traffic. From S9, action **a92_Reconnect** leads the system back to **S2_Balance_Requested**, rejoining the legitimate balance-view path. From S2, the system then proceeds through S3 and on to the normal terminal state S4 as previously described.

The scenario therefore induces four named execution paths. **ρ_legitimate** (S0 → S1 → S2 → S3 → S4) completes the balance view without disruption. **ρ_network-mitigated** (S0 → S5 → S6 → S8 → S9 → S2 → S3 → S4) and **ρ_server-mitigated** (S0 → S5 → S7 → S8 → S9 → S2 → S3 → S4) both recover from the DDoS attack via load-balancer mitigation and rejoin the legitimate path at S2. **ρ_criminal** (S0 → S5 → S6 or S7 → S8 → S10) represents the DDoS attack succeeding in cancelling the operation. The connection-loss path from S2 (via a28) constitutes a fourth entry into S8, meaning the criminal terminal S10 is also reachable from a user who was already mid-query when the attack disrupted the connection. All paths that reach S8 produce the same downstream check-level valuations from S8 onward, regardless of which disruption vector caused the disconnection.
