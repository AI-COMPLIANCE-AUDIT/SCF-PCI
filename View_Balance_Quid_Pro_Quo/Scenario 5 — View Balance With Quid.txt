**Scenario 5 — View Balance With Quid Pro Quo 3.1**

The scenario begins at **S0_Browser_Started**, the initial situation. From S0, two paths immediately diverge. The first is an abnormal termination: action **a021_Refusing_Connection_ABN** leads directly to **S25_Site_Maintenance_ABN**, an abnormal terminal state representing a refused connection at the entry point. The second path is the main flow: action **a01_Requesting** leads to **S1_Request_For_Connecting**, and from there action **a12_Connecting** leads to **S2_Connected**.

At **S2_Connected**, three events enter simultaneously — **E1_Receive_Bogus_Email_ABN**, **E2_Network_Problem_ABN**, and **E3_Specify_Credentials**. This state branches into three outgoing paths. The first involves a self-loop: the system cycles through action **a22_Checking_If_Website_Live**, returning to S2 until the site is confirmed live, after which action **a111_Refusing_Login** leads to **S16_Disconnected**, an abnormal terminal state representing login refusal. The second path is the legitimate flow: action **a23_Logging_In** leads to **S3_Credentials_Sent**. The third path is the social-engineering sub-graph: action **a212_Duping_Into_ABN** leads to **S12_IT_Support_Imitated_By_Hacker_ABN**, which is addressed separately below.

At **S3_Credentials_Sent**, events **E4_Credentials_Rejected** and **E5_Verify_Credentials** both enter. Two paths diverge. The first leads to the abnormal terminal state **S22_Invalid_Credentials** via action **a222_Sending_Credentials**, triggered when credentials are rejected. The second path proceeds via action **a34_Verifying_Credentials** to **S4_Logged_In**, at which point the item **SessionID (xsd:string)** enters scope confirming the authenticated session.

From **S4_Logged_In**, the system proceeds sequentially along the legitimate balance-view path. Action **a45_Choosing_Operation** leads to **S5_Operation_Chosen**, at which point the item **AccountID (xsd:string)** enters scope. Action **a56_Requesting_Balance** leads to **S6_Balance_Requested**, at which point event **E6_Return_Balance** enters and the item **CurrentBalance (xsd:int)** enters scope. Action **a67_Showing_Balance** leads to **S7_Balance_Shown**, at which point event **E9_User_Logout** enters. Action **a78_Logging_Out** leads to the normal terminal state **S8_Logged_Out**.

Returning now to the social-engineering sub-graph that branched from S2. At **S12_IT_Support_Imitated_By_Hacker_ABN**, both threats are active: **T2_Social_Engineering** and **T1_Quid_Pro_Quo**, where T1 is a subclass of T2. The quid pro quo attack is realised through IT support impersonation at this state. Action **a113_Developing_Into_ABN** leads to **S13_Machine_Overtaken_By_Hacker_ABN**, at which point two events enter — **E7_Reinstall_OS** and **E8_Credentials_Stolen_Directly_ABN**. From S13, two paths diverge.

The first path from S13 is the corrective recovery path: action **a133_Recovering_PC_Harddrive_COR**, triggered by **E7_Reinstall_OS**, leads to **S33_Normal_State**, representing OS reinstallation and system recovery. From S33, action **a35_Choosing_Operation** re-enters the legitimate path at **S5_Operation_Chosen**, and the system continues through S6, S7, and S8 as described above.

The second path from S13 is the direct access path: action **a135_Allowing_ABN** bypasses the corrective step entirely and leads directly to **S5_Operation_Chosen**, representing the hacker gaining access to the balance-view operations through the overtaken machine without requiring OS reinstallation. The system then continues through S6, S7, and S8 in the same manner as the legitimate path.

The scenario therefore induces five named execution paths. **ρ_refused** (S0 → S25) terminates at the entry point. **ρ_disconnected** (S0 → S1 → S2 → S16) terminates after login refusal. **ρ_invalid** (S0 → S1 → S2 → S3 → S22) terminates on credential rejection. **ρ_legitimate** (S0 → S1 → S2 → S3 → S4 → S5 → S6 → S7 → S8) completes the full balance-view flow. **ρ_qpq-reinstall** (S0 → S1 → S2 → S12 → S13 → S33 → S5 → S6 → S7 → S8) and **ρ_qpq-direct** (S0 → S1 → S2 → S12 → S13 → S5 → S6 → S7 → S8) both terminate at S8 after the social-engineering attack, diverging only at S13 based on whether the OS is reinstalled before the hacker proceeds. Both quid pro quo paths rejoin the legitimate flow at S5 and produce the same downstream check-level valuations from S5 onward.
