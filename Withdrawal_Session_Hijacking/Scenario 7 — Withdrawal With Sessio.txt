Scenario 7 — Withdrawal With Session Hijacking 3.1
The scenario begins at S0_Card_Inserted, the initial situation. At this state, event E4_Enter_PIN enters and the item CardID (xsd:int) enters scope, confirming that a physical card has been inserted into the ATM. Action a01_Entering_PIN leads to S1_PIN_Entered.
At S1_PIN_Entered, event E1_Check_PIN enters, representing the ATM verifying the entered PIN. From S1, two paths diverge. The first is a rejection cycle: action a10_Rejecting_ABN leads back to S0_Card_Inserted, representing an incorrect PIN causing the ATM to return to the card-inserted state and prompt the user to re-enter their PIN. The second path is the successful authentication path: action a12_Authenticating leads to S2_User_Authenticated.
At S2_User_Authenticated, the threat T1_Session_Hijacking enters, and two items enter scope simultaneously — SessionID (xsd:string) and CurrentBalance (xsd:int) — alongside AccountID (xsd:string). Event E2_Refuse_Card_ABN also enters at this state. From S2, two paths diverge. The first is the legitimate withdrawal path: action a23_Executing_Operation leads to S3_Cash_Withdrawn. The second is the session-hijacking criminal path: action a82_Stealing_ABN leads to S8_User_Identity_Stolen_ABN, which is addressed separately below.
At S3_Cash_Withdrawn, event E3_Request_Cash enters, and it is associated with the item Cash (xsd:int), representing the specific cash amount requested by the user. From S3, two paths diverge. The first is the successful clearing path: action a34_Clearing_Transaction leads to S4_Transaction_Successful, at which point event E7_Print_Receipt enters. Action a45_Removing_Card then leads to the normal terminal state S5_Card_Removed. The second path from S3 is the insufficient-funds path: action a19_Declining_ABN leads to S6_Insufficient_Amount, at which point the item Cash (xsd:int) enters scope at this state as the requested amount that could not be fulfilled. Action a65_Removing_Card then leads directly to the normal terminal state S5_Card_Removed, bypassing the transaction-clearing step.
Returning now to the session-hijacking path that branched from S2. At S8_User_Identity_Stolen_ABN, event E5_Credentials_Stolen_Directly_ABN enters, representing the attacker stealing the session credentials and user identity through the hijacked session. Action a89_Obtaining_ABN leads to S9_Payment_To_Criminals_ABN. At S9, event E6_Transfer enters, and two items enter scope — CurrentBalance (xsd:int) and AmountExtorted (xsd:int) — representing the financial data now under the attacker's control. Action a94_Transferring_Money leads to S4_Transaction_Successful, rejoining the legitimate path at the transaction-cleared state. At S4, event E7_Print_Receipt enters as before, and action a45_Removing_Card leads to the normal terminal state S5_Card_Removed.
The scenario therefore induces four named execution paths. ρ_legitimate (S0 → S1 → S2 → S3 → S4 → S5) completes the cash withdrawal successfully. ρ_insufficient (S0 → S1 → S2 → S3 → S6 → S5) terminates normally after a declined withdrawal due to insufficient funds. ρ_criminal (S0 → S1 → S2 → S8 → S9 → S4 → S5) represents the session-hijacking attack succeeding — the criminal completes a fraudulent transfer through the hijacked session and the system then proceeds to the card-removal terminal as if the operation were legitimate. The PIN-rejection cycle at S1 constitutes a sub-path within the entry phase that does not produce a distinct terminal; it loops until either successful authentication or card removal. The most forensically significant structural feature of this scenario is that the criminal path rejoins the legitimate path at S4, meaning a fraudulent transaction cleared via S9 is indistinguishable from a legitimate cleared transaction at the terminal state S5, and the check-level valuations from S4 onward are identical across both ρ_legitimate and ρ_criminal.
