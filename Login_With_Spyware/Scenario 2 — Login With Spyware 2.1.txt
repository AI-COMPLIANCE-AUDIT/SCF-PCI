Scenario 2 — Login With Spyware 2.1
The scenario begins at S0_Browser_Started, the initial situation. At this state, event E01_Send_URL enters, carrying the item Site_URL which is associated with it. Simultaneously, the threat T2_Spyware is present in the environment, and the item Antivirus_Software is linked to T2_Spyware — the antivirus is actively targeting the spyware throughout the scenario.
From S0, the system transitions to S1_Connection_Requested via two concurrent paths. The first is the direct path through action a01_Requesting_Site. The second is an indirect path that runs through the Antivirus_Software item: action a02_Running_Antivirus_Software engages the antivirus against T2_Spyware, followed by a02_Removing_Virus, after which the antivirus item leads the system to S1. Both paths arrive at S1.
At S1_Connection_Requested, event E13_Connect enters. E13 is associated with the item Site_URL, which in turn is linked back to the prior event E01. When E13 fires, the system loops through action a11_Reconnecting, cycling back to S1 until a connection is established. Additionally, events E11_No_Connection and E12_File_Not_Found both enter S1 and, when they occur, the system transitions to S2_Connected via action a12_Loading_Default_Page.
At S2_Connected, event E21_Navigate enters. E21 is associated with the item Page_URL. When E21 fires, the system loops through action a21_Reloading, cycling back to S2 until the correct page loads. Events E12_File_Not_Found (carried over from S1) and E22_Disconnect also enter S2. The main forward transition from S2 is to S3_Credentials_Sent via action a22_Navigating_to_Login_Page.
At S3_Credentials_Sent, event E31_Enter_Credentials enters, and it is associated with the two items Provided_User_Name and Provided_Password. This state has two outgoing paths. The first path proceeds to S4_Credentials_Received via action a31_Submitting_Credentials. The second path — the abnormal one — leads to S45_Credentials_Stolen via action a42_Copying, where T2_Spyware intercepts a copy of the credentials. S45 is not a terminal state: action a43_Storing cycles the system back to S3, allowing the spyware to store the stolen credentials while the user continues their login attempt unaware.
From S4_Credentials_Received, there is also a return path to S2 via action a42_Rejecting_Credentials, representing a server-side rejection that sends the user back to the connected state to re-enter credentials. S4 has two further events entering: E41_Accept_Credentials, which is associated with the items Provided_User_Name and Provided_Password, and E42_Invalid_Credentials. When E41 fires, the system transitions to the terminal state S5_User_Logged_In via action a41_Authenticating, at which point the item Session enters scope. When E42 fires instead, the system transitions to the abnormal terminal state S6_Login_Refused via action a43_Denying_Access, representing access denial due to invalid credentials.
The scenario therefore induces three execution paths: ρ_successful (S0 → S1 → S2 → S3 → S4 → S5), ρ_denied (S0 → S1 → S2 → S3 → S4 → S6), and ρ_stolen (S0 → S1 → S2 → S3 → S45 → S3 → S4 → S5 or S6), where the spyware silently copies credentials during the S45 cycle before the user completes or fails their login.
