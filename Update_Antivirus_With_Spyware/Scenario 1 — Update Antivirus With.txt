**Scenario 1 — Update Antivirus With Spyware 5.1**

The scenario begins at **S0_Antivirus_Not_Updated_ABN**, the initial situation. This state represents a host system whose antivirus software has not been updated, creating an exposed starting condition. Action **a01_Creates_ABN** leads the system to **S1_Vulnerability_ABN**. At this point, event **E1_User_Login** enters, representing a user logging in to the vulnerable system. From S1, two paths diverge simultaneously, constituting the two infection branches of the scenario.

The first branch is the **email attachment path**. Action **a12_Develops_Into_ABN** leads from S1 to **S2_Infected_Attachment_Opened_ABN**. At S2, event **E2_Bogus_Email_Received_ABN** enters. The item **Port** is also present at this state, introduced via object property **d04_Opens_ABN**, which connects T2_Spyware to the Port — meaning the spyware opens a port on the host. The threat **T2_Spyware** is a subclass of **T1_Malware**, and T1_Malware is linked to the item **Antivirus_Software** via object property **d02_Run_COR**, representing the antivirus software that is supposed to protect against the malware but has not been updated. From S2, action **a23_Infects_ABN** leads to **S3_Infected_With_Malware_ABN**.

The second branch is the **baiting path**. Action **a122_Develops_Into_ABN** leads from S1 to **S22_Device_Loaded_Into_PC**. At S22, the threat **T3_Baiting** appears — a physical baiting attack in which a malicious device is loaded into the PC. Action **a222_Executes_ABN** leads to **S23_Infected_Software_Ran_ABN**, representing the infected software on the baiting device executing on the host. From S23, action **a233_Infects_ABN** leads to **S3_Infected_With_Malware_ABN**.

Both infection branches therefore converge at **S3_Infected_With_Malware_ABN**. At S3, event **E3_Pop_Up_Clicked** enters, which is the trigger that enables the downstream branching from this state. S3 has three outgoing paths.

The first path from S3 is the **direct corrective path**: when event **E4_Run_Antivirus** enters instead of E3, action **a45_Malware_Removed_COR** leads directly to the normal terminal state **S5_Normal_State**, representing successful malware removal without further compromise.

The second path from S3, enabled by **E3_Pop_Up_Clicked**, leads via action **a333_Results_In_ABN** to **S332_Keystrokes_Captured_ABN**. At S332, event **E5_Run_Antivirus** enters. Action **a335_Malware_Removed_COR** then leads to the normal terminal state **S5_Normal_State**, representing a corrective recovery after keystrokes have been captured but before the machine is fully overtaken.

The third path from S3, also enabled by **E3_Pop_Up_Clicked**, leads via action **a34_Results_In_ABN** to **S4_Machine_Overtaken_By_Hacker_ABN**, which is an abnormal terminal state representing the hacker achieving full control of the machine with no subsequent recovery.

The scenario therefore induces four named execution paths. **ρ_corrective-direct** (S0 → S1 → S2 → S3 → S5) and its symmetric equivalent via the baiting branch (S0 → S1 → S22 → S23 → S3 → S5) both terminate at the corrective normal state when antivirus is run at S3 before E3 fires. **ρ_keylog-corrective** (S0 → S1 → S2 or S22/S23 → S3 → S332 → S5) terminates at S5 after keystrokes have been captured. **ρ_criminal** (S0 → S1 → S2 or S22/S23 → S3 → S4) terminates at the criminal abnormal state. The two infection branches — the email attachment path reaching S2 and the baiting path reaching S22 and S23 — are structurally symmetric up to S3 and produce the same downstream check-level valuations from S3 onward for all requirement checks evaluated in this scenario.
